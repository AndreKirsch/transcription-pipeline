## High-Level Flow

```mermaid
flowchart LR
  A[Dropbox
  Recordings] --> B[Fetch Audio
  pipeline/fetchAudio]
  B --> C[tmp_audio/]
  C --> D[Transcribe
  pipeline/transcribe]
  D --> E[tmp_transcripts/]
  E --> F[Store
  pipeline/store]
  F --> G[(MongoDB
  call_transcripts)]
```

1. **Fetch Audio (`pipeline/fetchAudio.ts`)**
   - Lists Dropbox files under the configured root path, filters by extension and excluded folders.
   - Downloads new files to `tmp_audio/` and returns metadata used downstream.
   - Skips already-downloaded content when `only_new_files` is enabled.

2. **Transcribe (`pipeline/transcribe.ts`)**
   - Reads audio from the fetch step (or local `step.input` fallback).
   - Calls OpenAI’s transcription endpoint with configurable prompt templating.
   - Emits transcript records (JSON) to `tmp_transcripts/` for durability.

3. **Store (`pipeline/store.ts`)**
   - Loads transcripts produced in-memory or from disk.
   - Applies `document_template` substitutions and bulk inserts into MongoDB.

## Configuration Layer

- **`transcription_workflow.yaml`** defines connections, schedule, and ordered steps.
- **`src/config/index.ts`** loads YAML + `.env`, resolves `${ENV}` placeholders, and validates the result with Zod before caching.
- Schema typing is enforced via `src/types/workflow.d.ts`; runtime validation ensures misconfigurations fail fast.

## Services

- **Dropbox Service** wraps the SDK with listing/downloading helpers and normalization routines.
- **OpenAI Service** streams files to the GPT transcription API, supports prompts, and returns verbose results.
- **Mongo Service** manages connections, batched inserts, and the processed-file ledger (unique index on `dropbox_id` + `content_hash`) used for idempotency.

### Idempotency Ledger

- Implemented via MongoDB collection (default `processed_files`).
- Keys: Dropbox file ID + content hash (or revision) prevent duplicate transcription/storage.
- Upsert occurs post-ingestion; ledger is consulted before downloads to skip previously processed files.

### MongoDB Index Strategy (planned)

- Primary key: autogenerated `_id`.
- Recommended secondary indexes once schema stabilizes:
  - `{ filename: 1 }` for quick lookup by original file name.
  - `{ created_at: -1 }` or `{ metadata.serverModified: -1 }` for recency queries.
  - Compound indexes for analytical needs (e.g., `{ dropbox_path: 1, created_at: -1 }`).
- Align index creation with migration scripts to avoid performance regressions.

## Utilities

- **`logger.ts`** – Pino JSON logger with run/session IDs and structured bindings.
- **`file.ts`** – filesystem helpers (ensure directories, read/write JSON, list files).
- **`templates.ts`** – lightweight `{{variable}}` templating for prompts and Mongo documents.
- **`constants.ts`** – canonical paths and defaults; ensures consistency when resolving directories.

## Orchestrator (`src/index.ts`)

- Loads workflow config, instantiates services, and tracks pipeline context (`fetchedFiles`, `transcripts`).
- Iterates steps, delegating to specific executors based on `action` value.
- Provides fallback behaviors:
  - Transcribe step loads from local audio if no Dropbox downloads occur.
  - Store step reads transcripts from disk when not available in memory.
- Ensures directories exist before use and closes Mongo connections via `finally`.

## Extensibility

- Step registry is currently switch-based; future plan is to map `action` strings to handler modules for dynamic loading.
- Additional services (e.g., S3, queue processor) can mimic the existing `services/*` patterns.
- Prompt templating already supports context injection (`file.*`, `now`). Document templates reuse same utility.

## Logging & Observability Plan

- Structured Pino logging with `run_id`, `step`, and file metadata (implemented).
- Next steps: emit per-step metrics and integrate with external log aggregation/alerting.

## Testing Strategy (Planned)

- **Unit Tests**
  - Mock Dropbox/OpenAI/Mongo services to validate pipeline step behavior.
  - Template substitutions and path resolution helpers.

- **Integration Tests**
  - Run against Docker Compose stack with local MongoDB and fixture audio files.
  - Assert transcripts are created and stored as expected.

- **End-to-End Smoke**
  - Scheduled in staging: run full workflow with synthetic audio, verify pipeline health.

## Security Considerations

- Avoid storing secrets in repo or logs; rely on `.env` for local and secret manager for prod.
- Ensure temp directories are cleaned or rotated to prevent sensitive data accumulation.
- Add transport security when communicating with external services (SDKs already use HTTPS).

## Future Enhancements

- Config schema migrations/versioning (validation implemented; migrations pending).
- Job scheduler abstraction (cron, queue, event-driven triggers).
- Retry/backoff and dead-letter handling for failure scenarios.
